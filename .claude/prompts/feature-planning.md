# Feature Planning Template — MVPTemplate

Используй этот шаблон для планирования новой фичи.

## 1. Описание фичи

### User Story
```
КАК [роль пользователя]
Я ХОЧУ [действие]
ЧТОБЫ [ценность/результат]
```

### Контекст
- Зачем эта фича нужна?
- Какую проблему решает?
- Как влияет на бизнес-метрики?

## 2. Acceptance Criteria
```
- [ ] AC1: Пользователь может ...
- [ ] AC2: Система должна ...
- [ ] AC3: При ошибке ...
- [ ] AC4: На мобильном ...
```

## 3. Технический дизайн

### Frontend (если есть UI)
- Какие экраны/компоненты затронуты?
- Новые компоненты в packages/ui/?
- Навигация (новый route в apps/mobile/app/)?
- Состояние (новый store в packages/store/)?
- Анимации?
- Platform.OS различия?

### Backend (если есть API)
- Новые/изменённые endpoints?
- Новые/изменённые таблицы БД?
- Миграции Drizzle?
- Бизнес-логика в service?
- Валидация (Zod schemas)?

### i18n
- Новые ключи переводов?
- Все 4 локали обновлены?

## 4. Тест-план

### Unit-тесты (Vitest)
- [ ] Test case 1: ...
- [ ] Test case 2: ...
- [ ] Edge case: ...

### E2E тесты (Playwright)
- [ ] Happy path: ...
- [ ] Error path: ...
- [ ] Mobile viewport: ...

## 5. Зависимости
- Новые npm-пакеты?
- Изменения в конфигурации?
- Feature flags?
- Связь с подпиской/оплатой?

## 6. Чеклист перед реализацией
- [ ] User story и AC согласованы
- [ ] Технический дизайн определён
- [ ] Тест-план готов
- [ ] Миграции БД спланированы
- [ ] i18n ключи определены
- [ ] Feature flag нужен? (packages/template-config)
- [ ] Onboarding: нужен шаг wizard / coach mark?
- [ ] Pricing: free или premium фича?

## 7. Порядок реализации (TDD)
1. Написать тесты (RED)
2. Backend: schema → repository → service → routes
3. Frontend: store → компоненты → экраны
4. i18n: все 4 локали
5. E2E тесты
6. Code review
7. Обновить контекст (context-keeper)
